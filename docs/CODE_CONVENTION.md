# 코드 컨벤션 — 클린 코드 원칙

이 문서는 로버트 C. 마틴(Robert C. Martin)의 「클린 코드」에 기반하여, 본 저장소(`DataPersistence`)와 이를 사용하는 프로젝트(`DataMonitor`, `DummyDataGenerator`, `SampleOrderSystem` 등)에서 코드를 작성할 때 따라야 할 원칙을 정리한다.

## 1. 가독성이 최우선이다

- 읽기 쉬운 코드가 좋은 코드다. 실행 속도나 코드 줄 수를 줄이는 것보다, 다른 사람(또는 미래의 자신)이 빠르게 이해할 수 있는 코드를 우선한다.
- 영리하고 압축적인 한 줄 코드보다, 여러 줄로 풀어 쓰더라도 의도가 분명한 코드를 택한다.

## 2. 이해하기 쉬운 코드가 곧 유지보수가 좋은 코드다

- 코드를 변경/확장하기 쉬운지가 품질의 기준이다. 처음 작성할 때 이해하기 쉽게 쓰는 것이 이후 버그 수정과 기능 추가 비용을 낮춘다.
- 주석으로 복잡함을 설명하기보다, 코드 자체의 구조(이름, 함수 분리)로 의도를 드러낸다.

## 3. 이름에는 의미가 있어야 하며, 함수는 한 가지 역할만 담당한다

- 변수/함수/클래스 이름은 그 자체로 "무엇을 하는지", "무엇을 담고 있는지" 설명해야 한다. 축약어나 의미 없는 이름(`data`, `tmp`, `flag2`)을 피한다.
- 함수(메서드)는 하나의 작업만 수행한다. 함수 이름에 "그리고(and)"가 들어가야 한다면 함수를 분리해야 한다는 신호다. 예를 들어 리포지토리의 `save()`는 파일에 쓰는 일만 하고, 검증이나 ID 부여는 별도 함수로 분리한다.
- 함수의 인자 수는 최소화하고, 부수효과(side effect)는 함수 이름에서 예측 가능해야 한다.

## 4. 코드는 글처럼 읽히도록 하고, 하나의 문서처럼 취급한다

- 위에서 아래로 읽었을 때 자연스러운 흐름(상위 개념 → 하위 개념)이 되도록 함수/클래스를 배치한다.
- 코드가 스스로 설명하지 못하는 부분에만 주석을 남긴다 — "무엇을 하는지"가 아니라 "왜 이렇게 했는지"(숨겨진 제약, 우회 이유, 놀라운 동작)만 기록한다. 예: 왜 임시 파일 → rename 방식을 쓰는지, 왜 롤백이 필요한지.
- 코드 포맷(들여쓰기, 공백, 파일 내 섹션 구성)을 일관되게 유지하여 문서처럼 스캔하기 쉽게 만든다.

## 5. 클래스는 하나의 책임을 갖는다 (SRP, Single Responsibility Principle)

- 클래스를 변경해야 할 이유는 오직 하나여야 한다.
- 본 저장소의 데이터 모델(`Model/Sample.h`의 `Sample`, 직렬화 책임) / 리포지토리(`SampleRepository`, JSON 파일 입출력 책임) / 콘솔 UI(`ConsoleApp`, 입출력 흐름 책임) 계층 분리(`docs/feature/json-parsing.md`, `docs/feature/json-file-storage.md`, `docs/design/phase0-foundation.md` 참고)는 SRP를 구조적으로 강제하기 위한 장치다 — `Sample`은 자신을 어떻게 직렬화할지만 알고, `SampleRepository`는 파일을 어떻게 읽고 쓸지만 알며, `ConsoleApp`은 사용자 입력을 어떻게 리포지토리 호출로 변환할지만 안다.
- 클래스가 여러 책임을 가지게 되면(예: 리포지토리가 콘솔 출력까지 겸하거나, `Sample`이 파일 경로를 알고 있는 경우) 즉시 분리한다.

## 6. 테스트 코드는 필수이며, 중복은 제거한다

- 새로 작성하는 기능에는 대응하는 테스트 코드를 함께 작성한다(경계값과 실패 경로 포함 — `.claude/agents/tester.md` 참고).
- 자주 반복되는 코드 패턴(중복)은 함수/클래스로 추출하여 제거한다. 단, 우연히 비슷해 보이는 코드까지 성급하게 추상화하지 않는다 — 반복이 실제로 같은 개념을 의미할 때만 통합한다.
- 리팩터링 시 테스트가 안전망 역할을 하므로, 테스트 없는 코드에 대한 리팩터링은 지양한다.

## 7. 적용 시 유의사항

- 이 원칙들은 phase 0~4 전 구간에 동일하게 적용된다. phase가 진행될수록 리포지토리/콘솔 UI에 기능이 늘어나더라도 원칙(가독성, SRP, 테스트)은 그대로 유지한다.
- 원칙 간 충돌이 있을 경우 가독성(1번)을 최우선 기준으로 판단한다.
